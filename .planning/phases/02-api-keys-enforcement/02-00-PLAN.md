**Title:** ADR 004: API Keys and Enforcement Architecture

**Status:** Accepted

**Context:**
- The proxy currently accepts only the master key on /v1/* — any client with the master key can use any model with no budget or rate controls.
- Phase 2 introduces per-application API keys that allow each application to have its own scoped credential with configurable model allowlists, rate limits (RPM/RPD), and hard budget caps.
- The enforcement must not materially increase per-request latency. Every /v1/* request must be validated without a synchronous DB lookup on the hot path.

**Decision: Schema** (covers D-01, D-02, D-03, D-04):
- Drop the placeholder api_keys table (no production data) and recreate with full Phase 2 schema.
- New api_keys table fields: id, application_id FK → applications, name, key_prefix, key_hash, max_rpm, max_rpd, max_budget (NULL = unlimited), soft_budget (NULL = none), is_active, created_at.
- Model allowlists in a separate key_allowed_models table (key_id FK, model_name TEXT). Normalized and queryable; empty set = all models allowed.
- usage_logs already has api_key_id FK — retain it, update FK target to new api_keys table.

**Decision: Key Format and Storage** (covers D-12, D-13):
- Format: sk-app-{48 hex chars}. The sk-app- prefix makes proxy keys visually distinct from upstream provider keys.
- Full plaintext returned once at creation; backend stores only key_prefix (first 8 chars for display) and SHA-256 hash. Key is never retrievable after creation.
- SHA-256 chosen over bcrypt: key lookup is on the hot path and SHA-256 is deterministic and fast; the long random key (24 random bytes = 192 bits of entropy) provides the security, not the hash function's computational cost.

**Decision: Enforcement Middleware** (covers D-05, D-06, D-10):
- /v1/* accepts both master key and per-app keys. Master key bypasses all enforcement (admin/testing credential).
- New KeyAuth middleware replaces Auth() on /v1/*. KeyAuth loads the full key record + allowlist and injects it into context (D-06). Downstream handlers read from context — no extra DB lookups per proxied request.
- Enforcement response codes: model not in allowlist → 403 permission_error; rate limit exceeded → 429 rate_limit_error; hard budget exceeded → 429 budget_limit_error; missing/invalid key → 401.

**Decision: In-Memory Enforcement Engine — keystore package** (covers D-07, D-08, D-09):
- New internal/keystore/ package with three components:
  1. Cache (cache.go): TTL-based key cache (default 60s). On revoke, immediately invalidated (not TTL-expired). Interface-agnostic design — future Redis backend can be swapped without changing callers.
  2. RateLimiter (counters.go): sync.Map of atomic int64 counters, keyed by (keyID, window). Window is current UTC minute (RPM) or current UTC day (RPD). Counters reset on restart — acceptable for single-instance deployment. Design does not preclude future Redis-backed distributed rate limiting.
  3. SpendAccumulator (spend.go): per-key running total initialized from SUM(total_cost) in usage_logs at startup. AcceptSpend() checks current total against hard budget without a DB read. Periodically flushed to storage (default 30s interval). Source of truth is the DB; accumulator is a performance optimization.
- All three are in-process, stdlib-only. No new external dependencies.

**Decision: Cost Attribution** (covers D-11):
- Cost is recorded after a successful response, including streaming (applied at stream completion).
- logRequest() extended to accept APIKeyID *int64. Nil = master key request.
- After logging, sa.Credit() updates the in-memory spend accumulator.

**Decision: Spend Flush Loop** (covers D-09):
- Periodic goroutine in main.go flushes the in-memory accumulator to usage_logs every 30s (configurable).
- On shutdown, final flush is performed before process exit.
- Flush implementation: iterate accumulator totals, INSERT or UPDATE usage_logs records per key.

**Decision: Frontend** (covers D-14, D-15, D-16, D-17):
- Keys view organized as Team → App → Keys drill-down, consistent with Phase 1 Applications view.
- Key creation form: name, model allowlist, RPM, RPD, hard budget, soft budget (all optional except name).
- Full key shown once in a modal with copy-to-clipboard; dismissing clears key from memory.
- Keys list shows prefix, name, model count, spend vs budget, active/revoked status, revoke action.

**Consequences:**
- Every /v1/* request now passes through an additional middleware layer (KeyAuth). Expected overhead: one sync.Map lookup + one atomic increment + one float64 comparison — sub-microsecond.
- Spend accumulator means spend totals are eventually consistent (up to 30s lag). Acceptable for Phase 2; hard budget enforcement uses the accumulator so a burst of requests within the flush window could briefly exceed the budget by the cost of one batch. This is an accepted tradeoff.
- Counter state lost on restart. RPM/RPD enforcement resets on restart — acceptable for single-instance small team use.
- Key revocation is immediate via cache.Invalidate() but in a future multi-instance deployment, other instances' caches would not be invalidated until their TTL expires. This is a documented limitation of the current single-instance design.

**Implementation files:**
- internal/storage/storage.go — APIKey struct, new Storage interface methods
- internal/storage/sqlite/migrations.go — Migration 11-13
- internal/storage/sqlite/apikeys.go — SQLite implementations
- internal/keystore/cache.go, counters.go, spend.go — Enforcement engine
- internal/api/middleware/keyauth.go — KeyAuth middleware
- internal/api/handler/keys.go — Admin key management handlers
- cmd/proxy/main.go — keystore initialization, spend flush loop goroutine
